# Дизайн-система @shadcnblocks в Claude — подробное руководство по использованию

Простыми словами: что это, как достать код, как вставить в проект, как понять «хороший ли блок». В конце — тренировочные промты, чтобы пощёлкать и разобраться на практике.

---

## 0. Что вы вообще подключили (в двух словах)

shadcnblocks — это **большая библиотека готовых кусков интерфейса** (hero-секции, ценники, футеры, формы, таблицы — всего ~3700 штук). Вы их не рисуете с нуля, а берёте готовый код и вставляете к себе.

Claude через MCP-коннектор работает как **продавец-консультант в этом магазине**:
- знает весь ассортимент,
- находит нужное по описанию,
- достаёт код любого блока,
- объясняет, как его поменять и куда вставить.

Вы общаетесь обычным текстом в чате. Никаких команд наизусть учить не нужно.

---

## 1. Как устроена работа — 4 действия

Всё, что умеет коннектор, сводится к четырём вещам:

| Действие | Что делает | Пример фразы |
|---|---|---|
| **Список** | показывает, что есть в каталоге | «покажи все категории в @shadcnblocks» |
| **Поиск** | находит блок по смыслу | «найди hero с картинкой справа» |
| **Показать код** | выдаёт исходник блока | «покажи код @shadcnblocks/hero2» |
| **Команда установки** | даёт готовую строку для терминала | «дай команду установки @shadcnblocks/hero2» |

### Формат имени блока
Всегда так: `@shadcnblocks/имя-номер`
Примеры: `@shadcnblocks/hero2`, `@shadcnblocks/pricing3`, `@shadcnblocks/navbar12`.

Блоки **пронумерованы**. Если сказать просто «hero» без номера — Claude выберет вариант сам. Хотите конкретный — называйте номер.

---

## 2. ФОРМАТ РАБОТЫ — как правильно просить

Чем точнее просьба, тем лучше результат. Хорошая просьба обычно содержит: **что за блок + как выглядит + под что.**

Плохо: «дай hero»
Хорошо: «покажи hero из @shadcnblocks: слева текст и две кнопки, справа скриншот приложения, тёмная тема»

### Мини-правила
- **Не уверены в номере** → сначала поиск, потом код. («найди…», выбрали, «покажи код @shadcnblocks/hero7»)
- **Нужен именно конкретный блок** → называйте имя с номером.
- **Собираете целую страницу** → просите «дай единым файлом», иначе получите куски вразнобой.
- **Всегда полезно** добавить «и скажи, какие зависимости нужны» — иначе код вставится, но может не запуститься.

## 3. СОХРАНЕНИЕ — как достать и сохранить код

Есть 4 способа забрать код себе. Выбор зависит от того, что дальше делаете.

### Способ A — просто скопировать из чата (самый простой)
Попросили «покажи код @shadcnblocks/hero2» → Claude выводит код → жмёте «копировать» и вставляете куда нужно. Годится, когда блок один.

### Способ B — Claude сохраняет в файл
«сохрани код @shadcnblocks/hero2 в файл hero2.tsx»
Claude создаст файл, который можно скачать. Удобно, если блоков несколько.

### Способ C — скачать напрямую через терминал (мимо Claude)
```
curl -H "Authorization: Bearer sk_live_ВАШ_КЛЮЧ" https://www.shadcnblocks.com/r/hero2 -o hero2.json
```
Получите сырой JSON с кодом. Полезно для скриптов и массовой выгрузки.

### Способ D — сразу установить в проект (см. раздел 4)
Это не «сохранить в сторону», а сразу разложить файлы по проекту. Делается через Claude Code.

> **Важно понимать разницу.** В этом чате-хабе Claude **показывает и отдаёт код**. Он НЕ раскладывает файлы в ваше приложение автоматически — для этого нужен раздел 4.

## 4. ПРИМЕНЕНИЕ — как вставить блок в реальный проект

Здесь ключевой момент, который путает почти всех:

**Чат-хаб = витрина (смотреть и брать код).
Установка в приложение = отдельная работа через Claude Code в папке проекта.**

Почему так: команда установки кладёт файлы в *текущую папку*. У хаба текущая папка — сам хаб, а не ваше приложение. Файлам физически нужно куда-то лечь — значит, нужно быть в папке приложения.

### Путь 1 — через Claude Code (рекомендуется)
1. Откройте Claude Code в папке вашего проекта (например IronMemo).
2. Убедитесь, что там есть `components.json` с тем же блоком `@shadcnblocks` и ключ (в `.env.local` проекта).
3. Попросите: «поставь @shadcnblocks/hero2 в проект».
4. Claude выполнит `npx shadcn add @shadcnblocks/hero2` и разложит файлы правильно.

### Путь 2 — вручную скопировать
1. В хабе: «покажи код @shadcnblocks/hero2».
2. Скопировали → создали файл в проекте, вставили.
3. Спросили «какие зависимости нужны» → доставили пакеты (`npm i ...`).
4. Проверили импорты (пути `@/components/...` должны совпадать с вашим проектом).

### Что почти всегда нужно доделать после вставки
- **Зависимости.** Блок может тянуть иконки, графики (recharts) и т.п. Без них — ошибка.
- **Тема и цвета.** Блоки идут в стиле `new-york` + Tailwind + CSS-переменные. Если у вас другая палитра — попросите Claude подогнать под ваши переменные.
- **Тексты и картинки.** Замените рыбу на свой контент.

### Полезные промты применения
- «дай команду установки @shadcnblocks/pricing3»
- «какие npm-пакеты нужны для @shadcnblocks/data-table12?»
- «подгони @shadcnblocks/navbar4 под мою тему: базовый цвет фиолетовый, скругления мягкие»
- «в @shadcnblocks/hero7 замени текст на: [ваш заголовок и подзаголовок], оставь одну кнопку»

## 5. ОЦЕНКА — как понять, хороший ли блок и подходит ли он

Не берите первый попавшийся. Перед вставкой прогоните блок по 6 вопросам — можно прямо спросить у Claude.

1. **Адаптивность.** «этот блок нормально выглядит на телефоне?» — многие блоки заточены под десктоп.
2. **Зависимости и вес.** «сколько доп. пакетов тянет этот блок?» — лишние тяжёлые зависимости раздувают проект.
3. **Доступность (a11y).** «есть ли в блоке проблемы с доступностью?» — важно для форм, меню, диалогов.
4. **Соответствие вашему стеку.** «этот блок подойдёт под Next.js + Tailwind v4?» — уточните вашу версию.
5. **Сравнение вариантов.** «покажи 3 варианта pricing из @shadcnblocks рядом, чем отличаются, какой лучше для SaaS» — выбирайте осознанно.
6. **Чистота кода.** «в этом блоке нет захардкоженных значений, которые мне придётся менять руками в 10 местах?»

### Быстрый чек-лист «брать / не брать»
- ✅ адаптивный, зависимостей мало, стиль близок к вашему, код читаемый → берём
- ⚠️ тянет 5+ пакетов ради мелочи, только десктоп, всё захардкожено → ищем другой вариант

*(один промт = поиск + оценка + код + применение + сохранение — если понятен этот шаг, вы освоили систему)*

## 7. Скрытые факторы (упускают почти все)

1. **«Показать» ≠ «установить».** В хабе Claude только показывает код и команды. Файлы в проект кладёт Claude Code в папке проекта. Не понимая этого, люди думают «коннектор не работает».
2. **Точность имени.** `hero2` и `hero125` — разные блоки. Без номера Claude выберет сам. Не уверены — сначала поиск.
3. **Зависимости — тихая ловушка.** Код вставится, но упадёт при запуске, если не доставить пакеты. Всегда спрашивайте про зависимости.
4. **Стиль под ваш проект.** Блоки в стиле `new-york` + свои CSS-переменные. Если у вас другая тема — вставленный блок будет выглядеть чужеродно, пока не подгоните переменные.
5. **Версия Tailwind/Next.js.** Часть блоков рассчитана на конкретные версии. Уточняйте свой стек, иначе классы могут не примениться.
6. **Единый файл против кусков.** Собирая страницу из нескольких блоков, обязательно просите «единым файлом» — иначе получите разрозненные фрагменты и потеряете время на склейку.

## 8. Альтернативные / «серые» варианты

- **Смотреть глазами в браузере.** Откройте shadcnblocks.com, найдите блок визуально, возьмите его имя (`hero125`) и в Claude просите код по имени. Часто быстрее, чем описывать словами.
- **Массовая выгрузка через curl.** Скриптом скачать пачку блоков напрямую (см. Способ C) — удобно для своей внутренней библиотеки.
- **Смешать реестры.** В `components.json` можно добавить и бесплатный `@shadcn`, и другие источники — тогда в одном промте искать сразу по нескольким («найди navbar в @shadcnblocks и @shadcn, сравни»).
- **Вшить навсегда.** Вытащенный код один раз сохранить в свой репозиторий — дальше не зависите от ключа и подписки. С оглядкой на условия лицензии.
- **Не зависеть от чата.** Если MCP капризничает — всё то же самое делается вручную: curl для кода + `npx shadcn add ./file.json` в проекте.
